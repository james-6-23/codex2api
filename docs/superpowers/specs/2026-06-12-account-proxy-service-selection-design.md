# 账号代理服务选择设计

日期：2026-06-12

## 背景

当前账号配置中的代理地址通过 `proxy_url` 字符串单独填写。代理管理页已经维护了代理服务列表，但账号新增/编辑表单无法直接选择这些代理服务，运维需要重复复制 URL，且代理管理中修改某个代理地址后，已配置账号不会自动跟随。

目标是将账号代理配置从手动填写 URL 优化为下拉选择代理管理中的代理服务，同时支持账号使用全部代理池/默认代理策略。

## 采用方案

采用方案 A：账号绑定代理服务 ID。

- 账号选择具体代理服务时，保存 `proxy_id`。
- 代理管理修改该代理服务 URL 后，绑定账号自动使用新 URL。
- 账号选择“全部代理池 / 默认代理策略”时，`proxy_id` 为空，使用现有代理池/全局代理/直连策略。
- 旧 `proxy_url` 字段保留，用于兼容历史自定义代理账号和外部 API 调用。

未采用方案：

- 保存 URL 快照：改动小，但不满足代理服务 URL 修改后账号自动跟随。
- 在 `proxy_url` 中保存 `proxy://<id>`：不新增字段但污染 URL 语义，容易破坏校验和 HTTP client 配置路径。

## 数据模型与兼容策略

### 数据库

为 `accounts` 表新增可空字段：

```sql
proxy_id INTEGER NULL
```

SQLite 和 PostgreSQL 初始化 schema、迁移补列都需要包含该字段。外键不强制约束，避免删除代理服务时阻塞操作；应用层负责识别缺失或禁用状态。

继续保留：

```sql
proxy_url TEXT/VARCHAR DEFAULT ''
```

### 优先级

运行时代理解析优先级：

1. `proxy_id` 指向的启用代理服务 URL
2. 历史 `proxy_url`
3. 全部代理池 / 全局代理 / 直连

如果 `proxy_id` 指向的代理服务不存在或已禁用，不自动降级到代理池，避免要求固定出口的账号悄悄换 IP。

### 响应字段

账号列表响应增加：

```json
{
  "proxy_id": 12,
  "proxy_url": "",
  "proxy_status": "bound"
}
```

`proxy_status` 可取值：

- `pool`：使用全部代理池/默认策略
- `bound`：绑定到可用代理服务
- `missing`：绑定的代理服务不存在
- `disabled`：绑定的代理服务已禁用
- `legacy_custom`：历史自定义 `proxy_url`

## 前端交互

### 字段替换

账号新增/编辑表单将“代理地址（可选）”替换为：

- 字段名：`代理策略`
- 默认选项：`全部代理池 / 默认代理策略`
- 具体代理服务选项：来自代理管理中的启用代理服务
- 历史兼容项：旧账号若存在手写 `proxy_url`，显示为 `历史自定义代理：<masked-url>`

### 选项行为

#### 全部代理池 / 默认代理策略

保存为：

```json
{
  "proxy_id": null,
  "proxy_url": null
}
```

运行时按现有系统策略：

- 代理池启用：使用全部启用代理池
- 代理池未启用但有全局代理：使用全局代理
- 都没有：直连

#### 指定代理服务

保存为：

```json
{
  "proxy_id": 12,
  "proxy_url": null
}
```

运行时实时解析 `proxies.id = 12` 的 URL。代理管理更新该服务 URL 后，账号自动跟随。

#### 历史自定义代理

仅用于旧账号兼容。

- 如果用户不切换，继续保留旧 `proxy_url`。
- 如果用户切换到代理池或代理服务，则清空旧 `proxy_url`。
- 新增账号界面不再提供自由输入，避免继续产生新的历史自定义代理。

### 测试代理

`测试代理` 按当前选择执行：

- 指定代理服务：测试该代理服务 URL。
- 历史自定义代理：测试旧 `proxy_url`。
- 全部代理池 / 默认代理策略：复用代理管理的一键测试逻辑。

全部代理池测试规则：

- 有启用代理服务：逐个测试启用代理，更新代理管理中的测试结果，返回汇总。
- 无启用代理但配置了全局代理：测试全局代理。
- 无任何代理配置：提示 `当前没有可测试的代理配置`。

推荐文案：

- `代理策略`
- `全部代理池 / 默认代理策略`
- `指定代理服务`
- `历史自定义代理`
- `测试代理`
- `代理池测试完成：成功 {{success}} 个，失败 {{failed}} 个`
- `当前没有可测试的代理配置`
- `当前绑定的代理服务不可用，请重新选择。`

## 后端 API 设计

### 新增账号

现有新增接口继续支持 `proxy_url`，新增支持 `proxy_id`。

场景：

- 全部代理池：`proxy_id: null`, `proxy_url: ""`
- 指定代理服务：`proxy_id: 12`, `proxy_url: ""`
- 外部兼容：仍可传 `proxy_url`

后端在新增/编辑时校验：

- `proxy_id` 非空时必须存在且启用。
- `proxy_id` 与 `proxy_url` 同时传入时，以 `proxy_id` 为准，并清空 `proxy_url`。

### 编辑账号

继续复用当前 `PATCH /accounts/:id/scheduler`，新增支持：

```json
{
  "proxy_id": 12,
  "proxy_url": null
}
```

语义：

- `proxy_id` 有值：绑定代理服务，并清空旧 `proxy_url`。
- `proxy_id: null` 且 `proxy_url: null`：切回全部代理池/默认策略。
- 历史自定义代理保持不变：不发送 `proxy_id/proxy_url` 或保留原 `proxy_url`。

### 代理测试

新增统一批量测试接口：

```http
POST /api/admin/proxies/test-all
```

响应：

```json
{
  "total": 10,
  "success": 8,
  "failed": 2
}
```

代理管理页和账号页都调用该接口，保证“一键测试”行为一致。

单个代理测试继续使用现有 `/proxies/test`。

## 错误处理与边界行为

### 代理服务不可用

当账号绑定 `proxy_id`，但代理服务被禁用或删除：

- 账号列表/编辑页显示 `当前绑定的代理服务不可用`。
- 运行时不自动降级到代理池。
- 测试代理时返回 `绑定的代理服务不存在或已禁用，请重新选择`。
- 保存编辑时要求选择其他可用代理服务，或切回全部代理池/默认策略。

### 代理管理变更影响

- 修改代理服务 URL：所有绑定该 `proxy_id` 的账号自动使用新 URL。
- 禁用代理服务：绑定账号变为不可用状态，不自动切换到代理池。
- 删除代理服务：绑定账号变为缺失状态，不自动切换到代理池。

### 向后兼容

- 外部 API 调用仍可传 `proxy_url`。
- 数据库旧字段不删除。
- 旧账号无需迁移即可继续使用。
- 本次不做“批量将历史自定义代理匹配到代理服务”的工具。

## 测试方案

### 后端测试

数据库迁移：

- 新库初始化包含 `accounts.proxy_id`。
- 旧库升级会自动添加 `proxy_id`。
- 旧 `proxy_url` 数据保持不变。

代理解析逻辑：

1. 账号绑定启用代理服务，返回该代理服务 URL。
2. 账号绑定禁用代理服务，不降级。
3. 账号绑定已删除代理服务，不降级。
4. 账号没有 `proxy_id`，但有历史 `proxy_url`，返回历史 URL。
5. 账号没有 `proxy_id/proxy_url`，保持现有代理池/全局代理逻辑。

管理接口：

- 新增账号传 `proxy_id`。
- 编辑账号切换代理服务。
- 编辑账号切回全部代理池。
- 编辑历史自定义代理不变。
- 绑定不存在代理 ID 时返回 400。
- 新增/编辑时只允许选择启用代理；已有绑定因后续禁用才进入异常状态。

代理池一键测试接口：

- 有启用代理：返回 `total/success/failed`。
- 无启用代理但有全局代理：测试全局代理。
- 无任何代理配置：返回可读错误。

### 前端验证

当前项目没有明确前端单测框架，使用类型检查和构建验证：

```bash
npm --prefix frontend run typecheck
npm --prefix frontend run build
```

手工验证：

1. 新增账号选择“全部代理池 / 默认代理策略”。
2. 新增账号选择某个代理服务。
3. 编辑账号从代理池切到指定代理。
4. 编辑账号从指定代理切回代理池。
5. 代理管理修改代理 URL 后，账号继续绑定同一个服务并使用新 URL。
6. 禁用/删除代理后，账号编辑页显示不可用状态。
7. `测试代理`：
   - 指定代理服务：测试单个 URL。
   - 全部代理池：触发一键测试并显示汇总。
   - 历史自定义代理：测试旧 URL。

## 验收标准

- 新增账号界面不再需要手动填写代理 URL。
- 运维人员可以从代理管理维护的代理服务中下拉选择。
- 账号可以选择单个代理服务，也可以选择全部代理池/默认策略。
- 绑定代理服务后，代理 URL 更新会自动生效。
- 代理不可用时不会静默降级，避免出口 IP 意外变化。
- 所有现有历史自定义代理账号继续可用。
